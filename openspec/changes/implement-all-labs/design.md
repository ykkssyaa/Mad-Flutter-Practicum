# Design: Единое приложение в корне проекта

Общий подход:
- Сделать один корневой Flutter-проект, который является итоговой реализацией всех лабораторных работ.
- Вынести общий код в корневую структуру `lib/` и не дублировать отдельные приложения по `lab_1`…`lab_6`.
- Ориентировать реализацию на Windows desktop: обеспечить запуск, нормальное масштабирование интерфейса и удобную навигацию в окне.
- Использовать современный Flutter-подход: четкое разделение на слои, небольшие переиспользуемые компоненты, понятный bootstrap приложения.
- Добавить базовые тесты для ключевых сценариев, а не дублировать smoke-тесты для каждой лабораторной папки.

Инфраструктурные решения:
- CI (опционально на этапе реализации): добавить шаблон GitHub Actions `.github/workflows/flutter.yml` для проверки корневого проекта на Windows-сборку, анализ и тесты.
- Зависимости: использовать минимально необходимые пакеты в едином `pubspec.yaml`; обновить версии только там, где это нужно для desktop/локализации/данных.
- Документация: обновить корневой `README.md`, чтобы он описывал запуск именно единого приложения и работу в Windows desktop.

Архитектурные направления по требованиям ЛР:
- Верстка и UI: выделить общий набор экранов, базовых виджетов и стилей из требований ЛР #1.
- Состояние: выбрать один понятный механизм управления состоянием для экранов и данных, чтобы не дублировать логику.
- Навигация: организовать единый маршрутный поток для desktop-окна с поддержкой переходов и обратной навигации.
- Данные: изолировать сетевые источники и локальное хранение в отдельных сервисах/репозиториях.
- Локализация: подключить локализацию на уровне приложения, а не на уровне отдельных папок.
- Desktop UX: предусмотреть поведение при ресайзе окна, адекватные размеры контента и базовую адаптивность.

Архитектурные замечания:
- Не переусложнять проект; достаточно ясного разделения на `app`, `features`, `data`, `domain`, `shared` или эквивалент.
- Считать лабораторные работами-источниками требований, а не отдельными приложениями.
- Следовать null-safety, устойчивой обработке ошибок и хорошему управлению состоянием.
