# Design: Как реализовать все лабораторные работы

Общий подход:
- Для каждой ЛР оставить её отдельную директорию (`lab_1`…`lab_6`). Работы будут реализованы внутри существующей структуры каждой папки.
- Сохранить исходный формат: каждая лабораторная работа должна иметь `lib/`, `assets/`, `pubspec.yaml`, `test/` и рабочий `lib/main.dart`.
- Там, где требуется — привести код в соответствии с современными best-practices Flutter (null-safety, разделение виджетов, небольшая архитектура: presentation / domain / data по необходимости).
- Добавить минимальный набор тестов: smoke widget test, и, где уместно, небольшие юнит-тесты.
- Настроить локализацию для ЛР6: `l10n.yaml`, ARB-файлы в `lib/l10n/`, и соответствующие `MaterialApp` локализационные делегаты.
- Проверки статического анализа: поправить `analysis_options.yaml` при необходимости и убедиться, что `flutter analyze` проходит без критических ошибок.

Инфраструктурные решения:
- CI (опционально на этапе реализации): добавить шаблон GitHub Actions `.github/workflows/flutter.yml` с задачами: checkout, setup Flutter, flutter pub get, flutter analyze, flutter test.
- Зависимости: использовать минимально необходимые пакеты. Если текущие `pubspec.yaml` уже указывают нужные — обновить версии при необходимости.
- Документация: в каждой `lab_n/README.md` или в корне `lab_n/` добавить краткие инструкции по запуску и требуемому SDK (например: Flutter 3.x/4.x), а также команды для тестов.

Примеры изменений по лабораториям (кратко):
- `lab_1`: проверить `lib/main.dart`, убедиться, что UI отображается; добавить примерные тесты для рендеринга виджета.
- `lab_2`: убедиться, что состояние управляется корректно; если используются устаревшие подходы, предложить замены (например, Provider/ValueNotifier).
- `lab_3`: проверить навигацию — настроить RouteGenerator/Router если требуется; добавить интеграционный сценарий навигации.
- `lab_4`: обеспечить мокирование сетевых вызовов в тестах (использовать пакет `http` и `http_mock_adapter`/`mockito` по необходимости).
- `lab_5`: проверить работу с локальными данными — обеспечить миграции/инициализацию (например `shared_preferences`, `sqflite`).
- `lab_6`: реализовать полные строки локализации и проверить переключение языков.

Архитектурные замечания:
- Не навязывать одной архитектуры на все лаб — для учебных целей достаточно простых разделений: widgets / services / models.
- Следовать null-safety и хорошему управлению состоянием.
